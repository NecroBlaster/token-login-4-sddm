# Protocol specification

The token protocol uses JSON messages over a line-based USB serial transport.

The reference transport is USB CDC ACM, but the protocol itself is transport-neutral.

## Request

The host sends one JSON object per line:

```json
{
  "v": 1,
  "op": "sign",
  "user": "example_user",
  "service": "sddm-token",
  "challenge": "BASE64_CHALLENGE"
}
```

Fields:

| Field | Meaning |
|---|---|
| `v` | Protocol version. Currently `1`. |
| `op` | Operation. Currently only `sign`. |
| `user` | Local account name requested by the host. |
| `service` | PAM service. Expected value: `sddm-token`. |
| `challenge` | Base64-encoded random challenge generated by the host. |

## Successful response

```json
{
  "ok": true,
  "alg": "Ed25519",
  "signature": "BASE64_SIGNATURE"
}
```

## Error response

```json
{
  "ok": false,
  "error": "user_not_allowed"
}
```

## Signed message

The token signs this byte sequence:

```text
sddm-ed25519-token-v1\0user\0service\0challenge
```

In Python form:

```python
message = (
    b"sddm-ed25519-token-v1\x00" +
    user.encode("utf-8") +
    b"\x00" +
    service.encode("utf-8") +
    b"\x00" +
    challenge
)
```

The protocol prefix provides domain separation. It prevents signatures created for this login protocol from being directly reused in another context.

## Replay resistance

The host must generate a fresh random challenge for each authentication attempt. A minimum challenge length of 32 random bytes is recommended.
