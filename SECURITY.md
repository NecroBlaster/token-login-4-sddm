# Security policy

## Scope

This project implements an alternative SDDM login path using an external Ed25519 challenge-response token.

It does not replace the normal password login path unless the system administrator chooses to remove or alter the normal SDDM PAM configuration. The recommended deployment keeps the normal password path and token path separated.

## Do not publish secrets

Never commit or publish:

- token private keys;
- `/etc/ed25519-token-auth/private.key`;
- production public keys if you consider them sensitive;
- real SDDM logs;
- screenshots containing usernames, hostnames, IP addresses, serial numbers, or personal details;
- compiled SDDM binaries;
- backups from `/root` or `/etc`.

## Threat model summary

The token signs a challenge generated by the host. The host verifies the signature using the user's public key.

This protects against password disclosure during token login because no password is sent through the token path.

However:

- possession of the token may be enough to authenticate as an allowed user;
- this is not automatically multi-factor authentication;
- a compromised host can still ask the token to sign challenges;
- a compromised token private key breaks the security of token login;
- the physical token must be protected.

## Recommended operational safeguards

- Keep `/etc/pam.d/sddm` unchanged as a password fallback during testing.
- Test `/etc/pam.d/sddm-token` with the PAM test tool before restarting SDDM.
- Keep a TTY session available before restarting the graphical display manager.
- Keep backups of original SDDM binaries and greeter themes.
- Use a unique USB serial number per token.
- Do not reuse the token keypair for other protocols.

## Reporting vulnerabilities

For a public repository, configure GitHub private vulnerability reporting or provide a dedicated contact address before accepting public reports.
